# plots.json - מבנה הנתונים

קובץ זה הוא מקור האמת היחיד של המחשבון: רשימות היישובים, מספרי המכרזים ומספרי המגרשים בתפריטי הבחירה נגזרות ממנו בלבד - דבר אינו קשיח ברכיבים. הקובץ הופק אוטומטית מחוברות המכרז (ראו `EXTRACTION_REPORT.md` בשורש הפרויקט) - אין לערוך ידנית ללא אימות מול החוברות.

> **אין לשמור כאן עלות רכיב קרקע - וזו החלטה מכוונת.** עלות הקרקע משתנה בין רוכשים (הצעה, הנחות, תנאי הקצאה), ולכן נתון מתוך החוברת יהיה שגוי עבור מרבית הזוכים. המחשבון משתמש **אך ורק בעלות הקרקע שהמשתמש מזין**. הוזן `0` - התיק מנותב לבדיקה ידנית ואינו נופל חזרה לנתון שמור.

## מבנה-על

```jsonc
{
  "purchaseTaxRate": 0.06,             // שיעור מס הרכישה בחישוב הפנימי
  "overpaymentThresholdILS": 1500,     // סף החריגה בש"ח לניתוב מסכי התוצאה
  "developmentToleranceILS": 12000,    // סטייה מותרת בין הפיתוח שהמשתמש שילם לנתון שבחוברת
  "settlements": [ ... ]             // יישוב אחד לכל מכרז
}
```

## יישוב (settlement)

| שדה | משמעות |
|---|---|
| `id` | מזהה לטיני קבוע (משמש בקוד) |
| `name` | שם היישוב בעברית - מוצג בתפריט |
| `tenderNumber` | מספר המכרז הרשמי כולל קידומת מרחב (למשל `בש/297/2025`) |
| `neighborhood` | השכונה נשוא המכרז |
| `developmentCompletionRate` | **אחוז הביצוע המשוקלל** (0-1) - נמסר על ידי הלקוח |
| `developmentCompletionRateSource` | מקור אותו אחוז (בערד - אינו מודפס בחוברת) |
| `developmentCostsVatNote` | בסיס המע"מ של הוצאות הפיתוח, כמודפס |
| `devCostIndexBase` | תאריך בסיס ההצמדה של הוצאות הפיתוח |
| `sourceBooklet` | שם קובץ החוברת בתיקיית `assets/` |
| `sourcePages` | עמודי המקור בחוברת |
| `plots` | מערך המגרשים |

## מגרש (plot)

| שדה | משמעות |
|---|---|
| `plotNumber` | מספר המגרש (מחרוזת - מוצג בתפריט) |
| `areaSqm` | שטח המגרש במ"ר - מוצג למשתמש כדי שיוודא שבחר את המגרש הנכון |
| `unitCount` | מספר יחידות הדיור במגרש (2 = דו-משפחתי). `null` = נתון סותר/חסר |
| `isDuplex` | `true` כאשר `unitCount >= 2`; `null` כשמספר היחידות אינו ידוע |
| `developmentCostFullPlot` | הוצאות הפיתוח **למגרש המלא**, בש"ח, **כולל מע"מ**, כמודפס. המחשבון מחלק ב-`unitCount` |
| `sourcePages` | עמוד המקור של הוצאות הפיתוח |
| `needsManualFill` | `true` = נתון חסר/סותר; המחשבון מנתב את המגרש לתוצאת "נדרשת בדיקה פרטנית" ואינו מחשב |
| `notes` | פירוט הבעיה כשקיימת |

## הנוסחה שהמחשבון מיישם

```
עלות פיתוח מותאמת = developmentCostFullPlot ÷ unitCount    // 2 יח"ד → מחצית; יח"ד אחת → מלוא הסכום
רכיב פיתוח רלוונטי = עלות פיתוח מותאמת × developmentCompletionRate
שווי משוער לצורכי מס = עלות הקרקע שהמשתמש הזין + רכיב פיתוח רלוונטי
מס רכישה משוער     = שווי משוער × purchaseTaxRate          // 6%
הפרש               = מס ששולם בפועל − מס רכישה משוער
```

הפרש מעל `overpaymentThresholdILS` (1,500 ₪) מציג את מסך "פוטנציאל להחזר". **אף אחד מהמספרים האלה אינו מוצג בממשק** - הם נשלחים בליד בלבד.

## הצלבת הוצאות הפיתוח

המשתמש נשאל כמה שילם בפועל עבור הפיתוח, והסכום מוצלב מול **חלקו שלו** בהוצאות הפיתוח לפי החוברת:

```
סטיית הפיתוח = | הפיתוח ששולם על ידי המשתמש − עלות פיתוח מותאמת |
```

הבסיס להשוואה הוא **עלות הפיתוח המותאמת** (חלקו של הרוכש), ולא הסכום שלאחר הכפלה באחוז הביצוע: `developmentCompletionRate` קובע כמה מהפיתוח נכנס לשווי לצורכי מס, לא כמה הרוכש שילם לרמ"י בפועל.

| הסטייה | המשמעות |
|---|---|
| עד `developmentToleranceILS` (12,000 ₪) | פער סביר. הוצאות הפיתוח בחוברת צמודות למדד (`devCostIndexBase`), ולכן תשלום במועד אחר הוא תשלום בסכום אחר. החישוב ממשיך כרגיל והמשתמש רואה הודעה. |
| מעל `developmentToleranceILS` | נתוני המגרש אינם מתארים את העסקה הזו, וחישוב אוטומטי מעליהם ייתן תשובה בטוחה ושגויה → בדיקה פרטנית. |

המשתמש רואה הודעה מילולית בלבד. הסטייה עצמה פנימית (§14) - הצגתה תחשוף את עלות הפיתוח שבחוברת.

החישוב אינו מתבצע כלל (→ בדיקה ידנית) כאשר: נעשה שימוש בהקלה בשומה, עלות הקרקע שהוזנה היא `0`, סטיית הפיתוח חורגת מהסף, המגרש אינו ברשימה, או `needsManualFill` מסומן.

## הוספת מכרז עתידי

מוסיפים אובייקט יישוב חדש למערך `settlements` עם כל השדות שלעיל (או מריצים מחדש את סקריפט החילוץ עם החוברת החדשה). תפריטי המחשבון יתעדכנו אוטומטית.
